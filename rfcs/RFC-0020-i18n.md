# RFC-0020 — i18n: Typed Translations & Finite String Types

- **Status:** Implemented (M1 + M2). M1 = finite string types & interpolation
  containment; M2 = the `std/i18n` generator library on RFC-0021. See the
  as-landed notes at the end.
- **Depends on:** RFC-0003 (validated types + compile-time constant proof),
  RFC-0010 (import flavors — the JSON-Schema import precedent), RFC-0013
  (globals machinery), the regex DFA engine

> **Motivation.** Modeled on a production Nuxt module
> (libr.stream `modules/i18n`): translation keys autocomplete, validate at
> compile time, carry per-key typed arguments derived from ICU messages, and
> even dynamically-constructed keys check when their parts are finite. Vyrn
> can do all of it — and the dynamic-key case *better* than TypeScript,
> because validated strings are DFAs: containment is a product-automaton
> walk, not a combinatorial union expansion.

---

## M1 — Finite string types & interpolation containment (the flagship) — IMPLEMENTED

> **Implemented.** The automata primitives live in `vyrn-frontend/src/regex.rs`
> (`Dfa::is_finite` / `Dfa::enumerate` / `Dfa::complement`, and free
> `contains` = shortest-witness product with the complement, `intersect` =
> conjunction, `concat_language` = literal-runs-and-hole-DFAs assembly). The
> containment policy is `vyrn-frontend/src/finite.rs`: it recognises the parser's
> desugared `@concat`/`@str` interpolation chain, builds `L = lit0·H1·lit1·…`
> when every hole is a finite string type (or a constant), and proves `L ⊆ T`.
> The checker runs it at every value boundary alongside `prove_coercion`
> (`Checker::prove_string_interpolation`): non-containment is a compile error
> with the witness; a proof lets codegen skip the runtime validation entirely
> (`Gen::coerce_flow`), while the interpreter's check is a no-op on a proven value
> (so `interp == native == wasm` holds untouched). The pure-`value =~` conjunction
> rule keeps length/other predicates on the runtime path. LSP string-literal
> completion (`string_literal_completions`) offers a finite type's enumerated
> language at call-argument and annotated-`let` positions. See
> `examples/finitekeys.vyrn`.



A validated `String` type whose regex denotes a **finite language** (its DFA
has no cycle on any accepting path — trivially decidable) is a *finite
string type*. Two checker powers follow:

1. **Interpolation typing.** `"nav.\{s}.label"` where every hole has a
   finite string type is itself typed as a finite regular language (the
   concatenation of literal parts and hole languages).
2. **Containment instead of constancy.** Coercing a finite-language
   expression into a validated string type checks **L ⊆ T by DFA product**
   (A ⊆ B ⇔ A ∩ ¬B = ∅ — complement + product + emptiness on the existing
   DFA representation). Statically proven ⇒ no runtime check emitted (the
   consteval precedent). Not contained ⇒ a compile error carrying a
   **witness**: "`nav.settings.label` (a possible value of this
   interpolation) does not satisfy `TransKey`" — the product automaton
   produces the counterexample for free.

TypeScript checks `` `nav.${Section}.label` `` by expanding the cross
product of unions (slow at thousands of keys, hard-capped near 100k
combinations). The DFA route never enumerates: cost scales with automaton
size. This feature is independent of i18n and lands first.

**LSP:** when the expected type at a string literal is a finite string
type, complete the literal's content with the type's language (enumerate up
to a cap of ~1000, else no suggestions). This is what makes `t("` offer
every key.

## M2 — Translation imports

```
locales/
  en.json     { "home": { "title": "Welcome" }, "cart": { "items": "{count, plural, one {# item} other {# items}}" } }
  uk.json     ...same key tree...
```

```vyrn
import i18n { t, setLocale, TransKey, Locale, tCartItems } from "./locales"

fn main() -> Int64 {
    setLocale(Uk)
    print(t("home.title"))          // key checked at compile time (finite type)
    print(tCartItems(3))           // typed args, derived from the ICU message
    return 0
}
```

**SUPERSEDED dispatch (design revision).** An earlier draft made `i18n` a
compiler-known import flavor (the `import type` precedent). That was
rejected as a crutch: it bakes a domain into the language, and every new
format would mean compiler code. M2 is now **a library on RFC-0021
(generator imports)** — an ordinary Vyrn `gen fn` that reads the locale
directory at compile time (sandboxed, deterministic, cached, lockable) and
returns the synthesized module as Vyrn source:

```vyrn
import { i18n } from "github:vyrn/i18n@v1/gen"   // or a local ./gen module
import { t, TransKey, setLocale } from i18n("./locales")
```

Everything below describes what that generator emits; the ICU parsing, key
flattening, drift checking, and CLDR plural tables are Vyrn library code.
Cross-locale drift = the generator traps = a load diagnostic at the import
site. The `import type` schema flavor remains grandfathered and is itself
reimplementable as a generator later.

The generator synthesizes a module containing:

- **`type TransKey`** — the finite validated string type of every dotted
  key (drives `t()` checking and LSP completion).
- **`type Locale = | En | Uk | …`** from the file names, plus
  **`setLocale(l: Locale)` / `locale() -> Locale`** backed by a
  generator-owned global (a narrow carve-out from root-only module state
  for *synthesized* modules).
- **One typed function per key** — `tHomeTitle() -> String`,
  `tCartItems(count: Int64) -> String` — with parameters derived from the
  ICU message (`{name}` → `String`, `{n, plural, …}` → `Int64`,
  `{k, select, …}` → a finite string type of the branch names). Each
  function's `///` doc is the source-locale message, so **hover shows the
  translation** with zero LSP work.
- **`t(key: TransKey) -> String`** — dynamic lookup for **argument-less**
  messages (a constant or finite key naming an argument-taking message is a
  checker error; a dynamic key reaching one at runtime returns the raw
  pattern, documented).

**ICU compilation, not an ICU runtime.** Each message compiles to a
generated *Vyrn* function body — string interpolation plus `if`/`match`
for plural/select — so all three backends get it for free and parity is
inherited, not re-proven. CLDR cardinal plural rules are compiled in from a
starter table in the generator (en-like, Slavic uk/ru/pl, de/fr/es, ja, …);
an uncovered locale is a clear compile error. ICU `date`/`time`/`number`
skeletons are out of scope v1.

**Cross-locale parity (the drift-catcher).** All locale files must have
the **same key tree** and per-key argument sets — a mismatch is a compile
error listing missing/extra keys per locale. (The Nuxt module needs a
baseline file and a check script for this; here it is just what `import`
means.)

## Speed (the question asked)

Keys compile once into a trie-shaped DFA (size linear in total key
characters); each constant check is an O(key-length) walk; containment is
polynomial in automaton sizes. The checker is a single-pass Rust program —
the current corpus + 595 tests check in under a second, and ten thousand
keys are noise at that scale. Per-keystroke LSP re-analysis stays at that
speed; there is no `tsc`-style union re-elaboration to pay for.

## Out of scope

ICU date/time/number formatting, locale negotiation/fallback chains, lazy
locale loading, runtime locale file loading (locales are compiled in — a
dynamic story can follow the wire-contract path later), bidi/collation,
message extraction tooling.

## Implementation notes (M2, as landed)

M2 is `std/i18n.vyrn` — a single `export gen fn i18n(dir)` plus ~50 pure helper
functions, all ordinary Vyrn. The compiler contains no i18n, ICU, or CLDR
knowledge. `examples/i18ndemo.vyrn` + `examples/locales/{en,uk}.json` are the
fixtures and a normal three-way parity citizen; `compiler/vyrn-cli/tests/i18n.rs`
drives `emit-gen`/drift/unsupported through the real binary, and the pure JSON +
ICU readers are covered by `test` blocks inside `std/i18n.vyrn` (they take a
`String`, so no file I/O is needed to exercise them).

- **The whole reader is written in Vyrn.** There is no JSON-parse builtin for
  arbitrary nested data (`fromJson` is type-directed), so the generator carries a
  small recursive JSON reader (objects + strings + nesting; a number/bool/null/
  array value is a generation error) and an ICU-subset reader, both operating on
  `bytes(s) -> Array<UInt8>` with `stringFromBytes` to rebuild strings. This was
  deliberate dogfooding; the only friction worth recording (fed to the language
  backlog): **no `else if`** (write nested `else { if … }` — the emitted plural/
  select chains are nested too), **no string `<`/`>`** (a byte-wise `strGreater`
  helper does lexicographic order), and **string element math is `UInt8` while
  `s[i]` is `Int64`** (so all parsing runs over `bytes(s)` to stay in `UInt8`).
  No new `std/strings` helpers proved strictly necessary.

- **ICU quoting decision (v1).** An apostrophe toggles a literal span and `''` is
  a literal apostrophe, so `'{'` / `'}'` yield literal braces; braces are
  otherwise structural. Supported placeholders: `{name}` (String), `{n, plural,
  … other {…}}` with `#` = the number and optional `=N` exact matches, and
  `{k, select, … other {…}}`. `other` is required. `\u` JSON escapes are rejected
  (write locale files in raw UTF-8); nested plural/select inside a branch is a
  generation error (a `{name}` inside a branch is fine and contributes an arg).
  This is the honest-over-wrong line the RFC's motivation asked for.

- **CLDR cardinal plural table (compiled into the emitted `if`-chain, not an ICU
  runtime).** en/de/es/it = {one, other}; fr/pt = {one (covers 0 and 1), other};
  uk/ru/pl = {one, few, many, other} with the mod-10/mod-100 rules; ja/zh/ko =
  {other}. A locale used with a plural message but absent from the table is a
  generation error (`I18N_NO_PLURAL_RULE__locale_<tag>`). Each message compiles
  to a per-locale helper `t<Key>__<Locale>(args)`; the exported `t<Key>` matches
  `currentLocale` and dispatches. `#` lowers to `count.toString()`.

- **The module-state carve-out (the one authorized frontend change).** The RFC's
  `setLocale`/`locale()` need module state, but a top-level `let` is root-only
  (RFC-0013). The loader now exempts a module SYNTHESIZED BY A GENERATOR (its key
  is the `generated by …` banner) — it is instantiated on behalf of the root, so
  its `currentLocale` global is as singular as the root's own, and its globals
  are merged into the linked program (initialized before `main`). A hand-written
  imported module still errors. Covered by two loader regression tests; also
  documented in RFC-0021's sandbox section.

- **Drift & errors.** All locales must share one key set AND per-key argument
  signatures (name + kind + select branch names, order-independent). A mismatch,
  an unsupported value, or an uncovered plural locale fails generation. Vyrn has
  no message-carrying trap, so — following the `std/rpc` precedent — each issue
  rides a bare top-level identifier in the emitted source (`I18N_DRIFT__locale_
  uk__missing__home_subtitle`, one per line); the synthesized module fails to
  parse and the diagnostics, attributed to the generator call site, name every
  offender.

- **`t(key)` deviation.** `t(key: TransKey)` accepts every key; an arg-less key
  returns the current locale's literal, and a key that takes arguments (only
  reachable via a dynamic key) returns its raw source pattern, as the RFC allows.
  The RFC's stronger "a constant arg-taking key is a checker error" would need a
  separate arg-less-only finite type for `t`'s parameter; deferred in favor of
  the single `TransKey` that also drives interpolation containment and completion.
