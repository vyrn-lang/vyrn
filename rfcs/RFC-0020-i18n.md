# RFC-0020 ‚Äî i18n: Typed Translations & Finite String Types

- **Status:** Draft ‚Äî approved for implementation (after RFC-0018/0019 land)
- **Depends on:** RFC-0003 (validated types + compile-time constant proof),
  RFC-0010 (import flavors ‚Äî the JSON-Schema import precedent), RFC-0013
  (globals machinery), the regex DFA engine

> **Motivation.** Modeled on a production Nuxt module
> (libr.stream `modules/i18n`): translation keys autocomplete, validate at
> compile time, carry per-key typed arguments derived from ICU messages, and
> even dynamically-constructed keys check when their parts are finite. Vela
> can do all of it ‚Äî and the dynamic-key case *better* than TypeScript,
> because validated strings are DFAs: containment is a product-automaton
> walk, not a combinatorial union expansion.

---

## M1 ‚Äî Finite string types & interpolation containment (the flagship)

A validated `String` type whose regex denotes a **finite language** (its DFA
has no cycle on any accepting path ‚Äî trivially decidable) is a *finite
string type*. Two checker powers follow:

1. **Interpolation typing.** `"nav.\{s}.label"` where every hole has a
   finite string type is itself typed as a finite regular language (the
   concatenation of literal parts and hole languages).
2. **Containment instead of constancy.** Coercing a finite-language
   expression into a validated string type checks **L ‚äÜ T by DFA product**
   (A ‚äÜ B ‚áî A ‚à© ¬¨B = ‚àÖ ‚Äî complement + product + emptiness on the existing
   DFA representation). Statically proven ‚áí no runtime check emitted (the
   consteval precedent). Not contained ‚áí a compile error carrying a
   **witness**: "`nav.settings.label` (a possible value of this
   interpolation) does not satisfy `TransKey`" ‚Äî the product automaton
   produces the counterexample for free.

TypeScript checks `` `nav.${Section}.label` `` by expanding the cross
product of unions (slow at thousands of keys, hard-capped near 100k
combinations). The DFA route never enumerates: cost scales with automaton
size. This feature is independent of i18n and lands first.

**LSP:** when the expected type at a string literal is a finite string
type, complete the literal's content with the type's language (enumerate up
to a cap of ~1000, else no suggestions). This is what makes `t("` offer
every key.

## M2 ‚Äî Translation imports

```
locales/
  en.json     { "home": { "title": "Welcome" }, "cart": { "items": "{count, plural, one {# item} other {# items}}" } }
  uk.json     ...same key tree...
```

```vela
import i18n from "./locales"        // a DIRECTORY of <locale>.json files

fn main() -> Int64 {
    setLocale(Uk)
    print(t("home.title"))          // key checked at compile time (finite type)
    print(tCartItems(3))            // typed args, derived from the ICU message
    return 0
}
```

The import synthesizes a module (the JSON-Schema-import machinery, new
flavor):

- **`type TransKey`** ‚Äî the finite validated string type of every dotted
  key (drives `t()` checking and LSP completion).
- **`type Locale = | En | Uk | ‚Ä¶`** from the file names, plus
  **`setLocale(l: Locale)` / `locale() -> Locale`** backed by a
  generator-owned global (a narrow carve-out from root-only module state
  for *synthesized* modules).
- **One typed function per key** ‚Äî `tHomeTitle() -> String`,
  `tCartItems(count: Int64) -> String` ‚Äî with parameters derived from the
  ICU message (`{name}` ‚Üí `String`, `{n, plural, ‚Ä¶}` ‚Üí `Int64`,
  `{k, select, ‚Ä¶}` ‚Üí a finite string type of the branch names). Each
  function's `///` doc is the source-locale message, so **hover shows the
  translation** with zero LSP work.
- **`t(key: TransKey) -> String`** ‚Äî dynamic lookup for **argument-less**
  messages (a constant or finite key naming an argument-taking message is a
  checker error; a dynamic key reaching one at runtime returns the raw
  pattern, documented).

**ICU compilation, not an ICU runtime.** Each message compiles to a
generated *Vela* function body ‚Äî string interpolation plus `if`/`match`
for plural/select ‚Äî so all three backends get it for free and parity is
inherited, not re-proven. CLDR cardinal plural rules are compiled in from a
starter table in the generator (en-like, Slavic uk/ru/pl, de/fr/es, ja, ‚Ä¶);
an uncovered locale is a clear compile error. ICU `date`/`time`/`number`
skeletons are out of scope v1.

**Cross-locale parity (the drift-catcher).** All locale files must have
the **same key tree** and per-key argument sets ‚Äî a mismatch is a compile
error listing missing/extra keys per locale. (The Nuxt module needs a
baseline file and a check script for this; here it is just what `import`
means.)

## Speed (the question asked)

Keys compile once into a trie-shaped DFA (size linear in total key
characters); each constant check is an O(key-length) walk; containment is
polynomial in automaton sizes. The checker is a single-pass Rust program ‚Äî
the current corpus + 595 tests check in under a second, and ten thousand
keys are noise at that scale. Per-keystroke LSP re-analysis stays at that
speed; there is no `tsc`-style union re-elaboration to pay for.

## Out of scope

ICU date/time/number formatting, locale negotiation/fallback chains, lazy
locale loading, runtime locale file loading (locales are compiled in ‚Äî a
dynamic story can follow the wire-contract path later), bidi/collation,
message extraction tooling.
