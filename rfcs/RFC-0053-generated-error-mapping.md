# RFC-0053 — Lex/Parse Errors in Generated Code Map Back to Their Source

- **Status:** Implemented
- **Depends on:** RFC-0033 (origin maps + `OriginMaps::remap` — this lifts
  its documented limitation), RFC-0048 (`.vyx` script/page origins),
  RFC-0049 (owner discovery), RFC-0021 (the gen cache)
- **Evidence (user):** a template produced
  ``in generated by vyxPageThemed("./routes/index.vyx", "./theme.json") at
  generated by pagesThemed("./routes", "./theme.json") at
  n:/lang/examples/bin/server.vyrn: unexpected character '\'`` — a
  dead-end message naming a synthesized module and no line in any file the
  user can open. RFC-0033's as-landed notes state the cause plainly: only
  **check/movecheck** diagnostics are remapped; **lex/parse/load** errors
  inside generated code are "never lost" but never *mapped*.

---

## 1. Remap lex/parse/load diagnostics too

A `.vyx` template's expressions (`{{ … }}`, `:attr="…"`, `v-if="…"`,
`@event="…"`) are copied **verbatim** into the synthesized module, so a
character the lexer rejects lands in generated code — exactly the class of
error most likely to be caused by what the user just typed, and currently
the one with the worst message.

- **The origin map does not need a successful parse.** It is built by a
  line-scan over the retained `gen_source` (RFC-0033), so it is available
  even when that source fails to lex or parse. Build it (or retain it)
  **before/independently of** the lex/parse step so a failing module still
  has a map.
- Route lex/parse/load diagnostics from a synthesized module through the
  same `OriginMaps::remap` the checker's already use: the reported
  location becomes the originating `.vyx` `file:line:col`, with the
  generated location preserved as the existing `note:`.
- **The never-lose guarantee is unchanged**: an error on a line with no
  governing directive (generator glue, not user text) keeps its generated
  location plus the note. Better a precise-but-generated location than a
  wrong one.

Result: the message above becomes
`examples/bin/routes/index.vyx:34:19: unexpected character '\'`
with `note: generated by vyxPageThemed(…) line 118`.

## 2. Verify unsaved `.vyx` edits actually re-generate

While probing this, an **unsaved** buffer edit that injected a stray `\`
into a template expression produced **no diagnostic at all** — suggesting
the gen cache may serve a previously-cached module instead of
regenerating from the overlay. If confirmed, the editor is analyzing what
is on *disk* rather than what you are *typing*, which would make §1
useless for live editing (the error only appears after save).

- Determine whether an overlay edit to a `.vyx` invalidates the
  generation for its owner (the RFC-0021 cache key must incorporate the
  overlay content of the inputs it read, not just their on-disk bytes).
- If it does not: fix it, so a `.vyx` keystroke re-generates and the
  diagnostic tracks the buffer.
- If it already does: state plainly why the probe saw nothing (and pin
  the real behaviour with a test) — do not invent a fix for a
  non-problem.

## Verification (reproduce first, then fix)

1. **Reproduce the raw failure**: a `.vyx` whose template expression
   contains a stray `\` must produce the dead-end message *before* the
   change (CLI, on disk). Capture it verbatim.
2. **After**: the same input reports at the `.vyx` line/col, with the
   generated location as a note — via **both** `vyrn check` (CLI) and the
   LSP driven over stdio with the VS Code URI form (`file:///n%3A/…`),
   published against the `.vyx` URI so the squiggle lands in the user's
   buffer.
3. A generated-glue failure (no governing directive) still reports with
   its generated location + note — nothing lost.
4. §2: a driver test that edits a `.vyx` **via overlay only** (never
   touching disk) and asserts the new diagnostic appears.
5. Full suite + LSP + parity green, 0 warnings. **Rebuild + HASH-VERIFIED
   redeploy** (fresh == deployed, both reported).

## As landed

### Where the origin-map build moved

`build_origin_maps` (a post-pass over the finished `Vec<Module>`) is **gone**.
Its work now happens inside `loader::load_modules`'s `visit`, the moment a module
is entered: as soon as a synthesized module's text is in hand — **before** it is
lexed — `OriginMaps::add_module` scans it for `//@origin` directives. The scan
never needed a parse; it just used to run somewhere a failing module could not
reach, because `visit` returns `Err` at the lexer and the module is only pushed
to `modules` on the way out.

Consequently:

- `load_modules` threads `&mut OriginMaps` through `visit` and returns it in
  **both** arms — `Err((diags, origins))` as well as `Ok((modules, key, origins))`.
  On the error path it runs every diagnostic through `OriginMaps::remap` before
  returning, so lex/parse/load diagnostics take the exact same remap the checker's
  already did.
- `loader::load_with_origins` therefore returns
  `(Result<Program, Vec<Diagnostic>>, OriginMaps)` rather than
  `Result<(Program, OriginMaps), _>` — the maps come back whether or not the load
  succeeded. `symbols::analyze_linked` assigns `Analysis.origins` from it
  unconditionally, so a `.vyx` that stopped lexing still knows its owner and still
  answers forward (hover/completion) queries.
- `Diagnostic` gains `from_generated: bool`, set by `OriginMaps::remap` alone. It
  is how the LSP tells a load diagnostic that now names a REAL file (publish it
  against that file's URI, into `Analysis.remapped`) from an ordinary foreign-file
  one (`adopt_foreign` into the root at line 0). Previously the load-error arm
  adopted everything — which is precisely why §2's probe looked dead.
- `publish` in the LSP appends a diagnostic's `note` to its message. VS Code
  renders the message verbatim and has nowhere else to put it; without it a `.vyx`
  squiggle gave no way back to the generated line.

### Before / after

**CLI** (`vyrn check` on `examples/bin/server.vyrn`, a stray `\` inside
`{{ t.binRecent() }}` on line 34 of `routes/index.vyx`):

```text
BEFORE  generated by vyxPageThemed("./routes/index.vyx", "./theme.json") at generated by pagesThemed("./routes", "./theme.json") at server.vyrn:79:29: unexpected character '\\'

AFTER   routes/index.vyx:34:12: unexpected character '\\'
          note: in generated code generated by vyxPageThemed("./routes/index.vyx", "./theme.json") at generated by pagesThemed("./routes", "./theme.json") at server.vyrn:79:29 (see `vyrn emit-gen`)
```

**LSP** (stdio driver, VS Code URI form `file:///n%3A/lang/…`):

```text
BEFORE  published against server.vyrn, line 0:
          in generated by vyxPageThemed("./routes/index.vyx", "./theme.json") at
          generated by pagesThemed("./routes", "./theme.json") at
          n:/lang/examples/bin/server.vyrn: unexpected character '\\'
        published against routes/index.vyx: (nothing)

AFTER   published against routes/index.vyx, range line 33 char 11 (0-based) —
        the squiggle lands on the expression in the user's buffer:
          unexpected character '\\'
          in generated code generated by vyxPageThemed(…) at … server.vyrn:79:29 (see `vyrn emit-gen`)
        published against server.vyrn: (nothing)
```

The remapped column is the governing directive's origin column — the START of the
verbatim region (the expression), not the offending character. A generated line
carries generator-chosen indentation and wrapping, so the offset from the
generated column back into the input is not recoverable from the table; region
start is the honest answer, and it is the position RFC-0033 already used for
check diagnostics.

### §2 verdict — no cache bug; the probe was §1 in disguise

**The RFC-0021 gen cache already keys on overlay content.** A cache entry records
each input path with the hash it had at generation time, and a hit re-verifies
them via `current_input_hash` → `resolver.read` → `EditorResolver::read`, which
prefers an open buffer's live text over the file on disk. An unsaved `.vyx`
keystroke therefore misses the cache and re-generates. Nothing to fix.

The probe saw nothing for a different reason, confirmed empirically by driving a
binary built from the parent commit: the overlay edit **did** regenerate and
**did** produce the diagnostic — but the load-error arm of `analyze_linked` sent
it through `adopt_foreign`, publishing it against **`server.vyrn` at line 0** with
the dead-end banner text, while `routes/index.vyx` was published with **zero**
diagnostics. A user watching the `.vyx` buffer they were typing in saw a clean
file. §1 is the whole fix; §2 needed only pinning, which
`rfc53_unsaved_vyx_edit_regenerates_and_squiggles` now does (it breaks the `.vyx`
via `didChange` only and asserts the on-disk text is still the good one).

## Out of scope

Recovering from a lex error to report *several* generated-code errors at
once (first error is enough), mapping runtime traps back to `.vyx`,
changing what the generators emit, and pretty-printing generated source
in the diagnostic.
