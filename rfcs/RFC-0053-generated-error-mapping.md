# RFC-0053 — Lex/Parse Errors in Generated Code Map Back to Their Source

- **Status:** Draft (design locked)
- **Depends on:** RFC-0033 (origin maps + `OriginMaps::remap` — this lifts
  its documented limitation), RFC-0048 (`.vyx` script/page origins),
  RFC-0049 (owner discovery), RFC-0021 (the gen cache)
- **Evidence (user):** a template produced
  ``in generated by vyxPageThemed("./routes/index.vyx", "./theme.json") at
  generated by pagesThemed("./routes", "./theme.json") at
  n:/lang/examples/bin/server.vyrn: unexpected character '\'`` — a
  dead-end message naming a synthesized module and no line in any file the
  user can open. RFC-0033's as-landed notes state the cause plainly: only
  **check/movecheck** diagnostics are remapped; **lex/parse/load** errors
  inside generated code are "never lost" but never *mapped*.

---

## 1. Remap lex/parse/load diagnostics too

A `.vyx` template's expressions (`{{ … }}`, `:attr="…"`, `v-if="…"`,
`@event="…"`) are copied **verbatim** into the synthesized module, so a
character the lexer rejects lands in generated code — exactly the class of
error most likely to be caused by what the user just typed, and currently
the one with the worst message.

- **The origin map does not need a successful parse.** It is built by a
  line-scan over the retained `gen_source` (RFC-0033), so it is available
  even when that source fails to lex or parse. Build it (or retain it)
  **before/independently of** the lex/parse step so a failing module still
  has a map.
- Route lex/parse/load diagnostics from a synthesized module through the
  same `OriginMaps::remap` the checker's already use: the reported
  location becomes the originating `.vyx` `file:line:col`, with the
  generated location preserved as the existing `note:`.
- **The never-lose guarantee is unchanged**: an error on a line with no
  governing directive (generator glue, not user text) keeps its generated
  location plus the note. Better a precise-but-generated location than a
  wrong one.

Result: the message above becomes
`examples/bin/routes/index.vyx:34:19: unexpected character '\'`
with `note: generated by vyxPageThemed(…) line 118`.

## 2. Verify unsaved `.vyx` edits actually re-generate

While probing this, an **unsaved** buffer edit that injected a stray `\`
into a template expression produced **no diagnostic at all** — suggesting
the gen cache may serve a previously-cached module instead of
regenerating from the overlay. If confirmed, the editor is analyzing what
is on *disk* rather than what you are *typing*, which would make §1
useless for live editing (the error only appears after save).

- Determine whether an overlay edit to a `.vyx` invalidates the
  generation for its owner (the RFC-0021 cache key must incorporate the
  overlay content of the inputs it read, not just their on-disk bytes).
- If it does not: fix it, so a `.vyx` keystroke re-generates and the
  diagnostic tracks the buffer.
- If it already does: state plainly why the probe saw nothing (and pin
  the real behaviour with a test) — do not invent a fix for a
  non-problem.

## Verification (reproduce first, then fix)

1. **Reproduce the raw failure**: a `.vyx` whose template expression
   contains a stray `\` must produce the dead-end message *before* the
   change (CLI, on disk). Capture it verbatim.
2. **After**: the same input reports at the `.vyx` line/col, with the
   generated location as a note — via **both** `vyrn check` (CLI) and the
   LSP driven over stdio with the VS Code URI form (`file:///n%3A/…`),
   published against the `.vyx` URI so the squiggle lands in the user's
   buffer.
3. A generated-glue failure (no governing directive) still reports with
   its generated location + note — nothing lost.
4. §2: a driver test that edits a `.vyx` **via overlay only** (never
   touching disk) and asserts the new diagnostic appears.
5. Full suite + LSP + parity green, 0 warnings. **Rebuild + HASH-VERIFIED
   redeploy** (fresh == deployed, both reported).

## Out of scope

Recovering from a lex error to report *several* generated-code errors at
once (first error is enough), mapping runtime traps back to `.vyx`,
changing what the generators emit, and pretty-printing generated source
in the diagnostic.
